# Invoice API — .NET port
 
ASP.NET Core 8 minimal-API port of the Node.js [invoice-api](../invoice-api).
Same endpoints, same matching algorithm, same SQLite database, different stack.
 
Built to learn .NET and to demonstrate that the same domain logic can move
between languages cleanly.
 
## What it does
 
Identical to the Node version:
 
- **Invoices**: create, list (filtered + paginated), fetch one, update status
- **Payments**: record + auto-attempt three-tier matching
  1. **Reference match** (confidence 1.00) — auto-closes invoice
  2. **Exact amount + normalised name** (confidence 0.95) — auto-closes invoice
  3. **Fuzzy name + exact amount** (confidence 0.75–0.99) — flagged for human review
The `MatcherService` is a near-line-for-line port of the JS `matcher.js` —
same normalisation, same suffix list, same LCS-based similarity. Confidence
scores for the same input agree to three decimal places across both stacks.
 
## Run
 
```bash
dotnet restore
dotnet run
```
 
Then open http://localhost:5251/swagger.
 
Default API key is `dev-key-change-me`. Override in `appsettings.json` or via env:
 
```bash
ApiKey="my-secret-key" dotnet run
```

## What changed in translation, and why
 
| Concept              | Node version            | .NET version                    |
|----------------------|-------------------------|---------------------------------|
| Web framework        | Express                 | ASP.NET Core minimal APIs       |
| ORM / DB driver      | better-sqlite3 (raw SQL)| EF Core + LINQ                  |
| Validation           | Zod                     | FluentValidation                |
| Auth                 | Express middleware      | ASP.NET middleware              |
| OpenAPI              | Hand-written YAML       | Auto-generated by Swashbuckle   |
| DTOs                 | Loose objects           | Records with explicit shapes    |
| Transactions         | `db.transaction(...)`   | `db.Database.BeginTransactionAsync()` |
| Decimals             | `number` (lossy!)       | `decimal` (exact, currency-safe) |
 
The last row is the one that genuinely matters. JavaScript's `number` is a
64-bit float; comparing `1500.00 === 1500.00` is reliable for clean test data
but will eventually bite you on real money math (e.g. `0.1 + 0.2 !== 0.3`).
.NET's `decimal` is purpose-built for currency and stores amounts exactly.
For a finance/AR product this is a real correctness improvement, not just
a stylistic choice.
 
## Known limitations
 
Same as the Node version — partial payments, bundled payments, multi-currency,
and bank-file ingestion (CAMT.053 / MT940) are all out of scope for the prototype.
 
## Possible next steps
 
- Migrations via `dotnet ef migrations add Initial` instead of `EnsureCreated()`
- Per-client API keys stored hashed in the DB
- Bundled-payment matching (combinatorial search per customer)
- Replace LCS similarity with embedding-based name matching at scale
- Background job to re-attempt unmatched payments when new invoices arrive
